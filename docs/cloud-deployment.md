# ThreatLens AI — Cloud Deployment (Render + MongoDB Atlas)

Deploy ThreatLens AI to the cloud using **Render** for application services and **MongoDB Atlas** for the database (free tiers available).

## Architecture (cloud)

```
Browser
   │
   ▼
threatlens-frontend.onrender.com  ──HTTPS──▶  threatlens-backend.onrender.com
                                                      │
                                                      ├── MongoDB Atlas (MONGODB_URI)
                                                      └── threatlens-ai.onrender.com
```

## Prerequisites

- [GitHub](https://github.com) repo with ThreatLens AI pushed
- [Render](https://render.com) account (free tier works)
- [MongoDB Atlas](https://www.mongodb.com/atlas) account (free M0 cluster)

---

## Step 1 — MongoDB Atlas

1. Create a free **M0** cluster.
2. **Database Access** → add user with password.
3. **Network Access** → add `0.0.0.0/0` (allow from anywhere) for Render.
4. **Connect** → Drivers → copy connection string:
   ```
   mongodb+srv://USER:PASSWORD@cluster0.xxxxx.mongodb.net/threatlens?retryWrites=true&w=majority
   ```
5. Replace `<password>` with your user password (URL-encode special characters).

---

## Step 2 — Deploy on Render (Blueprint)

1. Go to [Render Dashboard](https://dashboard.render.com).
2. Click **New** → **Blueprint**.
3. Connect repository: `johnadel112/ThreatLens-AI`.
4. Render detects `render.yaml` and creates 3 services:
   - `threatlens-ai`
   - `threatlens-backend`
   - `threatlens-frontend`
5. Set **required secrets** when prompted:

| Variable | Service | Value |
|----------|---------|-------|
| `MONGODB_URI` | backend | Atlas connection string |
| `OPENAI_API_KEY` | ai | Optional — leave empty for fallback agents |
| `CORS_ORIGINS` | ai | Optional — backend/frontend URLs |

6. Click **Apply** and wait for builds (~5–10 min first time).

---

## Step 3 — Seed demo users

After `threatlens-backend` is live:

1. Open the backend service in Render → **Shell**.
2. Run:
   ```bash
   node scripts/seed-users.js
   ```

Demo logins:

| Role | Email | Password |
|------|-------|----------|
| Analyst | analyst@threatlens.local | Analyst123! |
| Admin | admin@threatlens.local | Admin123! |

---

## Step 4 — Load simulated events

From your local machine:

```bash
cd simulator
```

Create/update `simulator/.env`:

```env
BACKEND_URL=https://threatlens-backend.onrender.com/api
SIMULATOR_API_KEY=<copy from Render backend env>
```

```bash
npm run simulate:full-demo -- --instant
```

> Free-tier Render services spin down after inactivity. First request may take 30–60s to wake up.

## Service URLs

After deploy, Render provides URLs like:

| Service | URL pattern |
|---------|-------------|
| Frontend | `https://threatlens-frontend.onrender.com` |
| Backend | `https://threatlens-backend.onrender.com` |
| AI Service | `https://threatlens-ai.onrender.com` |

The frontend automatically points to the backend via `BACKEND_HOST` (set by the blueprint).

---

## Environment variables reference

### Backend (`threatlens-backend`)

| Variable | Required | Description |
|----------|----------|-------------|
| `MONGODB_URI` | Yes | MongoDB Atlas connection string |
| `JWT_SECRET` | Yes | Auto-generated by Render |
| `AI_SERVICE_URL` | Auto | Linked from AI service hostname |
| `CORS_ORIGINS` | Auto | Linked from frontend hostname |
| `SIMULATOR_API_KEY` | Auto | For event simulator API key auth |
| `NODE_ENV` | Yes | `production` |

### AI Service (`threatlens-ai`)

| Variable | Required | Description |
|----------|----------|-------------|
| `OPENAI_API_KEY` | No | Enables LLM agents; omit for fallback mode |
| `MODEL_NAME` | No | Default `gpt-4o-mini` |
| `CORS_ORIGINS` | No | Comma-separated allowed origins |

### Frontend (`threatlens-frontend`)

| Variable | Required | Description |
|----------|----------|-------------|
| `BACKEND_HOST` | Auto | Render sets this → runtime `config.js` |

## Troubleshooting

| Issue | Fix |
|-------|-----|
| CORS errors | Ensure `CORS_ORIGINS` on backend includes frontend hostname (Render links this automatically) |
| DB connection failed | Check Atlas IP allowlist and `MONGODB_URI` password encoding |
| AI investigation timeout | Free tier cold start — retry; or upgrade plan |
| Login 401 | Run `seed-users.js` in backend shell |
| Simulator 401 | Match `SIMULATOR_API_KEY` in simulator `.env` with backend |

## Alternative: Railway

Railway supports Docker deploys from the same Dockerfiles. Import the GitHub repo and create three services from `backend/`, `ai-service/`, and `frontend/` Dockerfiles with the same environment variables as above.

## Security checklist (before sharing publicly)

- [ ] Strong `JWT_SECRET` (Render auto-generates)
- [ ] Unique `SIMULATOR_API_KEY`
- [ ] Atlas user has minimum required permissions
- [ ] `OPENAI_API_KEY` only on AI service (never frontend)
- [ ] Rotate secrets if this is more than a demo deployment
